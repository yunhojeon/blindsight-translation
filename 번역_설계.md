# Blindsight 한국어 번역 — 작업 설계서

피터 와츠의 *Blindsight* (CC BY-NC-SA 2.5) 전문을 한국어로 번역해 온라인으로 공유하기 위한 절차·데이터 구조·UX 설계서다. 핵심 원칙은 **"번역 데이터"와 "표현(presentation)"을 완전히 분리**하는 것이다. 번역 결과는 구조화된 데이터로 저장되고, 독자가 보는 HTML은 그 데이터로부터 빌드 스크립트가 생성한다. 표현 방식을 바꿔도 번역을 다시 하지 않는다.

소스 분석 결과(설계의 전제):

- 분량 약 107,000 단어. 본문은 `<p>` 문단 약 4,200개로 구성된다.
- 챕터용 `<h2>`는 없다. 장면 구분은 가운데 정렬된 `*`, 부(部) 제목("The Book of Oogenesis" 등)은 가운데 정렬된 텍스트 줄로 표현된다.
- 이탤릭(`<i>`)이 약 2,200곳 — 강조와 내적 독백에 쓰인다.
- ConSensus/통신 장면은 오른쪽 정렬 문단(약 59곳) + 가운데 정렬 메타데이터 줄("encrypted", "Accept?" 등)로 표현된다.
- 원문에는 이미 와츠 본인의 미주(尾註) 144개("Notes and References")가 있다. 이것은 독자가 추가하려는 역주와 별개다.
- 라이선스는 CC BY-NC-SA 2.5 (저작자 표시·비영리·동일조건변경허락).

## 1. 전체 아키텍처

```
원본 HTML
   │  (1) 추출 — 1회성
   ▼
segments.jsonl   ← 본문을 안정적 id를 가진 segment로 분해 (불변)
   │
   │  (2) 번역 루프 — 청크 단위, 반복
   ├──────────────┐
   ▼              ▼
translations/     glossary.json + characters.json   ← 매 청크마다 주입·갱신
*.jsonl           annotations.json
   │
   │  (3) 빌드 — 표현 변경 시마다 자유롭게 재실행
   ▼
reader.html  (단일 self-contained, 온라인 공유용)
```

세 단계는 단방향이다. (2)의 산출물(번역·용어집·역주)만이 "정본 데이터"이고, (3)은 언제든 다시 돌려도 (2)를 건드리지 않는다. 폰트·색·원문 토글 방식·레이아웃을 바꾸려면 빌드 스크립트만 수정한다.

### 디렉토리 구조

```
blindsight-translation/
├─ source/
│  └─ Blindsight by Peter Watts.html      # 원본 (읽기 전용)
├─ data/

│  ├─ translations/

│  └─ annotations.json                     # 역주 (용어집과 별개의 자유 주석)
├─ build/
│  └─ build_reader.py                      # (3) 빌드 스크립트
├─ dist/

└─ 번역_설계.md
```

## 2. 데이터 모델

### 2.1 segments.jsonl — 본문의 골격 (불변)

본문을 **segment** 단위로 쪼갠다. 1 segment = 원문 1 문단(`<p>`)이 기본. segment는 한 번 추출하면 **id를 절대 바꾸지 않는다.** 번역·원문 토글·역주가 모두 이 id를 참조하므로, id 안정성이 분리 구조의 핵심이다.

id는 원문 등장 순서 기반의 4자리 0-패딩 정수(`s0001`, `s0002`, …)로 부여한다. 순서 기반이므로 사람이 위치를 직관적으로 파악할 수 있고, 청크 경계와도 잘 맞는다.

각 줄(JSON):

```json
{
  "id": "s0412",
  "kind": "para",
  "align": "justify",
  "part": "Theseus",
  "scene": 7,
  "text": "Szpindel looked up. \"From that?\"",
  "runs": [
    {"t": "Szpindel looked up. \"From "},
    {"t": "that", "i": true},
    {"t": "?\""}
  ],
  "watts_notes": ["sdfootnote12"]
}
```

| 필드 | 의미 |
|---|---|
| `id` | 불변 식별자 |
| `kind` | `para`(본문) / `scene-break`(`*`) / `part-title`(부 제목) / `comms-meta`(통신 메타 줄) / `epigraph` 등 |
| `align` | `justify` / `center` / `right` — 원문 정렬 보존 |
| `part`, `scene` | 부·장면 번호 (네비게이션·청크 경계용) |
| `text` | 원문 평문 (검색·표시·원문 토글용) |
| `runs` | 인라인 서식 보존. `i:true`는 이탤릭. 이걸로 "어디를 강조했나"를 번역에 전달 |
| `watts_notes` | 이 문단에 달린 와츠 원주 id (있으면) |

`runs`가 중요하다. 이탤릭은 단순 장식이 아니라 *어느 단어를 강조했는지*가 의미를 가지므로(특히 내적 독백·반어), 평문이 아니라 run 배열로 보존해 번역 단계에 그대로 넘긴다.

원주(와츠 미주)는 별도 파일 `data/watts_notes.json`에 원문/번역을 함께 보관한다.

### 2.2 translations/*.jsonl — 번역 (정본)

segment와 1:1 대응. 청크 파일로 쪼개 저장(병렬 작업·diff·재개에 유리).

```json
{
  "id": "s0412",
  "status": "translated",
  "runs": [
    {"t": "스핀델이 고개를 들었다. \"저"},
    {"t": "것", "i": true},
    {"t": "에서?\""}
  ],
  "speaker": "Szpindel",
  "addressee": "Bates",
  "register": "반말",
  "glossary_used": ["Szpindel"],
  "translator_note": null,
  "revision": 2
}
```

- `runs`로 번역하면 강조 위치가 한글에서도 보존된다(빌드 때 색 강조로 렌더).
- `speaker`/`addressee`/`register`는 존댓말 일관성 검증과 디버깅에 쓴다(§4).
- `status`: `untranslated` → `draft` → `translated` → `reviewed`. 재개 시 어디까지 했는지 즉시 파악.
- 번역은 segment의 평문을 절대 덮어쓰지 않는다. 원문은 항상 segments.jsonl에 남아 원문 토글로 제공된다.

### 2.3 glossary.json — 용어집 (단일 정본)

기존 CSV(`이름,설명,영문이름,종류,…`)를 구조화·확장한다. 이것이 **일관성의 심장**이다.

```json
{
  "Scrambler": {
    "ko": "스크램블러",
    "strategy": "transliterate",
    "type": "neologism",
    "aliases": ["scramblers", "scrambler's"],
    "note": "로르샤흐 내부의 외계 생명체. 지적 생명체는 아니다.",
    "note_level": "spoiler-safe",
    "first_seen": "s1893",
    "locked": true
  },
  "ConSensus": {
    "ko": "콘센서스",
    "strategy": "transliterate",
    "type": "neologism",
    "aliases": ["Consensus"],
    "note": "테세우스호 내 무선 통신망. 승무원이 머릿속에서 직접 접속한다.",
    "first_seen": "s0150",
    "locked": true
  },
  "phenotype": {
    "ko": "표현형",
    "strategy": "translate",
    "type": "science",
    "first_mark": {"en": true, "hanja": "表現型"},
    "note": "개체의 겉으로 드러나는 특성(육체·행동 등). 유전형(genotype)의 반대. 작중에선 '확장된 표현형'이 로봇팔 같은 신체 확장을 가리킨다.",
    "note_level": "science",
    "first_seen": "s0...",
    "locked": false
  },
  "blindsight": {
    "ko": "맹시",
    "strategy": "translate",
    "type": "science",
    "first_mark": {"en": true, "hanja": "盲視", "title_link": true},
    "note": "뇌의 시각영역(V1) 손상으로 생기는 시각 장애. 환자는 보이지 않는다고 보고하면서도 물체의 위치·운동을 맞힌다. 이 작품의 제목이기도 하다(국내 통용 제목 '블라인드사이트').",
    "note_level": "science",
    "first_seen": "s....",
    "locked": true
  },
  "Fireflies": {
    "ko": "반딧불이",
    "strategy": "translate",
    "type": "neologism",
    "coinage": true,
    "first_mark": {"en": true},
    "note": "갑자기 대기권에 나타났다 수십 초 만에 사라진 65,536개의 외계 물체. 실제 곤충이 아니라 붙여진 이름.",
    "first_seen": "s....",
    "locked": true
  }
}
```

- 키 = 영어 원형. `aliases`로 복수형·소유격·대소문자 변형을 잡아 빌드 때 자동 링크/주석 부착.
- `type`: `neologism`(작품 고유명사) / `proper`(인명·함선명) / `science`(과학용어).
- `strategy`: 번역어 결정 방식. `transliterate`(음차) / `translate`(의역) / `keep`(원어 그대로). §2.3.1 참조.
- `note`: 독자용 짧은 주석. `note_level`로 *과학 해설*과 *스포일러 주의* 항목을 구분(빌드에서 노출 정책 결정).
- `locked: true`는 "번역 확정, 변경 금지" 플래그. 새 청크 번역 시 잠긴 용어는 반드시 그대로 써야 한다.
- `first_seen`: 처음 등장한 segment id. "첫 등장 때만 주석 표시"와 "첫 등장 때만 원어 병기" 정책에 사용.

### 2.3.1 번역어 전략 — 음차 vs 의역, 원어 병기

기존 출간 번역본은 작품 고유명사를 무리하게 한국어로 옮겨 어감을 해친 사례가 많다(Scrambler→'훼방꾼', ConSensus→'교감', skimmer→'수영선수'). 그래서 다음 원칙을 둔다.

- **작품 고유명사·신조어(`neologism`)와 인명·함선명(`proper`)** 중 한국어로 옮기면 SF의 어감이 죽거나 어색해지는 것은 **음차(`transliterate`)**한다. 원어 발음을 한글로 적는다. 예: 스크램블러, 콘센서스, 스키머, 텔레매터.
- **실제 과학용어(`science`)** 로서 한국어 학술 표준어가 있는 것은 그 **표준 번역어를 유지(`translate`)**한다. 예: 맹시(blindsight), 감각질(qualia), 단속운동(saccade), 표현형(phenotype). 독자가 검색·학습할 수 있는 정착된 용어이기 때문.

**첫 등장 병기는 `first_mark`로 항목별 제어한다.** 음차 여부(`strategy`)와 별개로, 빌드가 `first_seen` segment에서 무엇을 덧붙일지 정한다.

- `first_mark.en: true` → 첫 등장 시 `한글(English)` 병기. 음차어(스크램블러(Scrambler))뿐 아니라, **한글 발음만으로 뜻이 안 통하는 의역어**도 켠다. 예: `위상(topology)`, `인지권(noosphere)`, `감각질(qualia)`. 정착·자명한 음차(테세우스 등)는 끈다.
- `first_mark.hanja: "漢字"` → 한자어인데 동음이의·추상적이라 한자가 뜻을 분명히 해주는 경우에만 첫 등장 시 `한글(漢字)` 또는 `한글(漢字, English)`로 병기. 예: `위상(位相, topology)`, `표현형(表現型)`. **필요한 항목에만** 넣어 남용하지 않는다.
- `first_mark.title_link: true` → blindsight 전용. 작품 제목과 같은 단어임을 알린다. 빌드는 첫 등장 시 `맹시(blindsight)`로 병기하고, 주석에 "이 작품의 제목, 국내 통용 '블라인드사이트'"를 단다. 독자가 본문의 '맹시'와 책 제목 '블라인드사이트'가 같은 개념임을 알게 된다.
- 모든 병기는 **빌드 단계에서 자동 생성**한다. 번역 `runs`에는 한글만 넣고, 빌드가 `first_seen` 일치 segment에서 `first_mark` 규칙대로 덧붙인다. 표기 정책을 바꿔도(매 등장 병기/병기 제거/한자 토글) 번역 데이터는 불변, 빌드만 다시 돌린다.
- 음차 표기는 외래어 표기법을 기본으로 하되 어감을 우선해 항목별로 확정하고 `locked`한다(예: Sarasti→'사라스티').

**의역한 고유명사의 첫 등장 색 구분(`coinage`).** 원문은 `Fireflies`처럼 일반 명사를 고유명사로 처음 쓸 때 이탤릭으로 "이건 진짜 곤충이 아니라 붙인 이름"임을 신호한다. 한글에서 '반딧불이'라고만 쓰면 이 신호가 사라진다. 그래서 `coinage: true`인 항목은 **첫 등장 시 강조색과는 다른 별도 색(고유명사색)으로 렌더**해, 이것이 일반 명사가 아니라 작중 명칭임을 시각적으로 표시한다. 이후 등장은 평범하게 둔다(또는 옵션). 음차어는 이미 낯선 표기라 신호가 내장돼 있어 `coinage`가 필요 없다.

CSV는 사람이 편집하기 편하니, **CSV ↔ glossary.json 변환 스크립트**를 둬서 기존 CSV를 가져오고 다시 내보낼 수 있게 한다.

### 2.4 characters.json — 인물 + 존댓말 매트릭스

§4에서 상술. 인물 메타 + 화자쌍별 말투 규칙.

### 2.5 annotations.json — 역주 (용어집과 별개)

용어집은 "단어"에 붙지만, 어떤 주석은 특정 문장·문맥에만 붙는다(번역 선택 해설, 과학 배경, 복선 등). 이는 segment id에 매단다.

```json
{
  "s0259": [
    {"anchor": "인지불능증", "text": "agnosia. where 경로는 온존하나 what 경로 손상으로 사물의 의미·종류를 인출하지 못하는 인지장애. '아내를 모자로 착각한 남자'로 유명.", "level": "science"}
  ]
}
```

## 3. 분할(Chunking) 전략

추출은 1회뿐이지만, **번역은 청크 단위로 순차 진행**한다.

- **청크 크기**: segment 기준 약 30~50개, 영어 1,500~2,500단어. 한 세션에서 맥락을 유지하며 번역하고 용어집을 갱신하기 좋은 크기.
- **경계는 장면(scene)을 존중**한다. 청크가 장면 중간에서 끊기지 않게 `scene-break` 직후에서 자른다. 대화 톤·말투 맥락이 장면 안에서 이어지기 때문.
- **청크 파일명 = segment id 범위**(`0381-0420.jsonl`). 위치가 자명하고 병합 충돌이 적다.
- 각 청크는 독립적으로 재번역·교정 가능(다른 청크 불변).

### 번역 루프 (청크 1개 처리 절차)

매 청크마다 아래를 반복한다. 이 루프가 "용어집을 매 단계 공유·갱신"이라는 요건의 구현이다.

1. **컨텍스트 로드**: `glossary.json`(특히 `locked` 항목) + `characters.json`의 말투 매트릭스 + 직전 청크의 마지막 몇 문단(연속성) + 해당 청크 segment들.
2. **번역**: 위 컨텍스트를 프롬프트에 주입해 청크를 번역. 잠긴 용어는 그대로, 인물 말투는 매트릭스대로. `runs`로 강조 위치 보존.
3. **신규 용어 추출**: 이 청크에서 처음 나온 고유명사·신조어·과학용어를 뽑아 후보로 제시. 번역어·주석을 정하고 `glossary.json`에 추가, 확정되면 `locked: true`.
4. **말투 검증**: 청크 내 모든 대화의 `speaker→addressee→register`가 매트릭스와 일치하는지 점검(§4). 새 화자쌍이 등장하면 매트릭스에 규칙 추가.
5. **커밋**: `translations/<범위>.jsonl` 저장, `glossary.json`·`characters.json` 갱신. `status` 갱신으로 진행률 기록.

> 도구화 권장: 위 1·5의 데이터 입출력은 작은 파이썬 스크립트로, 2~4의 판단은 번역가(또는 Claude)가 맡는 반자동 워크플로가 가장 견고하다.

## 4. 존댓말 / 반말 일관성

한국어 번역에서 가장 깨지기 쉬운 부분. **인물 관계로 말투를 미리 확정**하고, 전 작품에 걸쳐 강제한다.

### characters.json

```json
{
  "characters": {
    "Siri":     {"ko": "시리 키튼", "role": "화자·종합가"},
    "Sarasti":  {"ko": "주카 사라스티", "role": "선장·흡혈귀"},
    "Bates":    {"ko": "어맨다 베이츠", "role": "전투 담당"},
    "Szpindel": {"ko": "아이작 스핀델", "role": "생물학자"},
    "James":    {"ko": "수잔 제임스", "role": "언어학자(4인방)"}
  },
  "register_matrix": {
    "Siri->Sarasti":  {"register": "존댓말", "basis": "지휘관-부하"},
    "Sarasti->Siri":  {"register": "반말",   "basis": "지휘관-부하"},
    "Szpindel->Siri": {"register": "반말",   "basis": "동료, 친근"},
    "Siri->Szpindel": {"register": "반말",   "basis": "동료, 친근"},
    "Bates->Sarasti": {"register": "존댓말", "basis": "지휘 체계"}
  },
  "register_events": [
    {"after": "s2890", "pair": "Siri->Chelsea", "change": "존댓말→반말", "reason": "관계 진전"}
  ]
}
```

- **매트릭스는 방향성**을 가진다(A→B와 B→A가 다를 수 있다). 화자→청자 쌍마다 말투를 명시하고 근거(`basis`)를 적어 나중에 흔들리지 않게 한다.
- 영어 원문엔 존비 구분이 없으므로 이 매트릭스가 **추가된 번역 결정**이다. 처음 등장하는 쌍은 그때 정하고 기록한다.
- **변화 이벤트**(`register_events`): 관계가 바뀌어 말투가 변하는 지점을 segment id로 못박는다. 예컨대 회상 속 연인 관계의 진전. 그 전후로 빌드/검증이 다른 규칙을 적용.
- **검증 패스**: 각 번역 segment에 기록된 `register`가 매트릭스(+이벤트)와 어긋나면 플래그. 전체 번역 후 일괄 검증 스크립트로 일관성 깨진 곳을 잡는다.

특수 사례: 시리의 1인칭 서술(평어체 종결 — 문학 번역 관례), 사라스티의 비인간적 화법, 4인방(다중 인격)의 인격별 말투 차이 등은 매트릭스에 인물별 "서술/독백 스타일" 항목으로 별도 고정한다.

## 5. 주석 전략 — 3계층

세 종류의 주석을 **출처와 노출 정책으로 구분**한다. 이래야 visual clutter를 통제할 수 있다.

1. **와츠 원주 (author notes)** — 원문 미주 144개. 원문 그대로 + 번역을 함께 보관(`watts_notes.json`). 본문 위치에 작은 위첨자 번호로 표시, 클릭 시 펼침. *원작의 일부이므로 항상 제공.*
2. **역자 용어주 (glossary notes)** — `glossary.json`의 `note`. 해당 용어 **첫 등장**에만 표시하거나(기본), 독자가 용어에 hover/탭 하면 표시. 과학용어·고유명사 이해용.
3. **역자 문맥주 (context notes)** — `annotations.json`. 특정 문장의 번역 선택·복선·배경 해설. 가장 적게, 꼭 필요한 곳에만.

빌드 시 `note_level`(`science` / `spoiler-safe` / `translation`)로 묶어, 독자가 "과학 해설만", "역주 끄기" 등을 토글할 수 있게 한다. 스포일러가 될 수 있는 주석(`Scrambler`의 정체 등)은 첫 등장 시 핵심 정보를 가리고 후반부에서 전체 공개하는 정책도 `note_level`로 표현 가능.

## 6. 포맷팅 보존 매핑

| 원문 | segment 표현 | reader.html 렌더 |
|---|---|---|
| `<i>강조</i>` | `run.i = true` | 옅은 색 강조 `<span class="emph">` (이탤릭 대체) |
| 오른쪽 정렬(ConSensus 발화) | `align:"right"` | `.comms-right` (오른쪽 정렬 + 통신 스타일) |
| 가운데 메타("encrypted") | `kind:"comms-meta"` | `.comms-meta` (작은 글씨·가운데) |
| 가운데 `*` 장면 구분 | `kind:"scene-break"` | 장식 구분선/별표 |
| 부 제목 | `kind:"part-title"` | 제목 스타일 + 목차 앵커 |
| 와츠 원주 위첨자 | `watts_notes` | 위첨자 번호 + 펼침 |
| 의역 고유명사 첫 등장(Fireflies 등) | glossary `coinage:true` + `first_seen` | 강조색과 다른 **고유명사색** `<span class="coinage">` |
| 첫 등장 원어/한자 병기 | glossary `first_mark` | `맹시(blindsight)`, `위상(位相, topology)` 자동 부착 |

강조색(이탤릭 대체)과 고유명사색(`coinage`)은 **서로 다른 색**이어야 한다. 둘 다 본문에 등장하므로 충돌하지 않게 CSS 변수로 분리 관리한다(예: 강조=따뜻한 색, 고유명사=차분한 색). 둘 다 색 + 약한 보조 신호(밑줄 점선 등)를 함께 줘 색각 접근성을 확보한다.

이탤릭은 **옅은 색 강조**로 렌더(결정됨). 한글엔 이탤릭이 없으므로 색이 가장 깔끔하다. 단, 색만으로 구분하면 접근성(색각)에서 약하니, 색 + 아주 약한 자간/굵기 보조를 함께 주는 것을 권장한다. 색상은 빌드 CSS 변수 하나로 통제해 나중에 바꿔도 번역 불변.

ConSensus 통신 장면은 좌/우 정렬로 화자를 구분하는 원작 의도를 살려, 오른쪽 정렬 + 옅은 배경/모노스페이스 같은 "통신 UI" 스타일을 입힌다.

## 7. 표현(UX) 설계 — reader.html

빌드 산출물은 **단일 self-contained HTML**(CSS·JS·데이터 인라인). 친구에게 링크/파일 하나로 공유 가능, 서버 불필요.

### 원문 열람 — 문단별 인라인 펼치기 (결정됨)

- 평소엔 번역만 보인다. 각 문단(또는 그 묶음) 끝에 아주 작고 옅은 표식(예: 작은 `›` 또는 단락 호버 시 나타나는 핸들).
- 누르면 **그 문단의 원문이 바로 아래에 인라인으로 펼쳐진다.** segment id로 1:1 대응하므로 정확히 그 부분만 나온다.
- 다시 누르면 접힘. 기본 상태는 "전부 접힘"이라 몰입 독서 시 화면이 깨끗하다.
- 상단에 전역 토글("원문 보기 전체 켜기/끄기")도 둬, 대조 정독 모드와 몰입 모드를 한 번에 전환.

### Clutter 최소화

- 기본 화면 = 번역 본문만. 용어주·문맥주·원문은 전부 숨김 상태에서 시작.
- 상단/설정에 토글: `원문` · `역주(용어)` · `과학 해설` · `와츠 원주`. 독자가 원하는 만큼만 켠다.
- 용어주는 본문에 표식을 박지 않고(첫 등장 밑줄 정도만 옵션), **hover/탭 시 팝오버**로 띄우는 것을 기본으로 — 본문 흐름을 해치지 않는다.
- 사용자의 토글 선택은 `localStorage`에 기억(서버 없이).

### 네비게이션

- 부/장면 기반 목차(`part`, `scene`). 긴 분량이므로 진행률 표시·현재 위치 저장(이어 읽기).
- segment id 기반 앵커로 특정 문단에 딥링크 가능(친구끼리 "여기 봐" 공유).

### 빌드가 번역을 건드리지 않음 (요건의 핵심)

표현을 바꾸고 싶을 때(색 변경, 원문 토글을 팝업으로 교체, 2단 병렬 추가, 폰트 교체 등) **`build_reader.py`/CSS만 수정하고 다시 빌드**한다. `data/`의 번역·용어집·역주는 그대로다. 반대로 번역을 고치면 빌드만 다시 돌리면 표현에 반영된다.

## 10. 번역 실행 방식 — `claude -p` 스크립트 루프

청크 수가 많으므로(약 30~50개 segment × 수십 청크), 대화형 subagent보다 **스크립트가 청크를 돌며 `claude -p`(헤드리스 모드)를 호출하는 방식이 적합하다.** 이유: 각 호출의 입력 컨텍스트(용어집·말투 매트릭스·직전 청크 꼬리)를 정확히 통제할 수 있고, 출력이 JSON으로 표준화되며, 중단 후 재개가 쉽다.

```bash
for chunk in data/chunks/*.json; do
  out="data/translations/$(basename "$chunk" .json).jsonl"
  [ -f "$out" ] && continue          # 이미 한 청크는 건너뜀(재개)
  claude -p \
    --system-prompt "$(cat build/translate_system.md)" \
    --output-format json \
    < <(python build/make_prompt.py "$chunk") \
    | python build/parse_result.py "$out"
done
```

핵심 설계점:

- **구조화 입출력**: `make_prompt.py`가 (현 `glossary.json`의 `locked` 항목 + `characters.json` 매트릭스 + 직전 청크 마지막 몇 문단 + 이번 청크 segments)를 하나의 프롬프트로 조립한다. 시스템 프롬프트는 "출력은 지정 JSON 스키마(translations + new_terms + register_checks)만"이라고 못박는다. `parse_result.py`가 결과를 검증·저장하고 신규 용어를 glossary에 머지한다.
- **잠긴 용어 강제**: 프롬프트에 `locked` 용어 목록을 매번 주입해 일관성을 보장. 모델이 어긴 곳은 파싱 단계에서 자동 치환/플래그.

### 순차 의존성 문제와 2단계 전략 (권장)

순진하게 돌리면 청크 N+1이 청크 N에서 새로 확정된 용어집에 의존하므로 **완전 순차**가 되어 느리다. 이를 풀기 위해 2단계로 나눈다.

1. **1단계 — 용어집·말투 사전 구축(빠른 패스)**: 전체 청크를 *번역하지 않고* 훑으며 고유명사·신조어·과학용어 후보와 화자쌍만 추출한다(`claude -p`로 청크당 가볍게). 결과를 모아 **`glossary.json`과 `characters.json`을 먼저 확정·`locked`** 한다. 사람이 한 번 검토해 음차/의역·말투를 정한다.
2. **2단계 — 본 번역(병렬 가능)**: 용어집이 얼어 있으므로 청크 간 의존이 사라진다. 청크들을 **병렬로** `claude -p` 호출해 번역한다(직전 청크 꼬리는 연속성용으로만 주입). 처리량이 크게 오른다.

이 구조는 §1의 분리 원칙과도 맞는다 — 용어집·말투가 데이터로 먼저 고정되고, 번역은 그 위에서 독립적으로 생성된다.

### 검증

전 청크 번역 후 별도 스크립트로 일괄 검증한다: 잠긴 용어 위반, 말투 매트릭스 불일치(`register` 필드 대조), 미번역(`status`) segment, `runs`의 강조 개수 원문 대비 누락 등. 위반 목록을 뽑아 해당 청크만 재번역한다.

> 요약: **2단계(사전 용어집 고정 → 병렬 번역) + `claude -p` 스크립트 루프 + JSON 입출력 + 사후 검증**을 권장한다. 단일 순차 루프도 동작하지만 느리고, 후반 청크일수록 좋은 번역이 나오는 대신 초반 용어가 흔들릴 수 있어 2단계가 더 안정적이다.

## 11. 독자 상태 — 이어 읽기 · 북마크 · 하이라이트 · 위치 공유

장편이라 한 번에 못 읽으므로, reader.html이 **읽은 위치·북마크·하이라이트를 기억**하고 **특정 위치를 URL로 공유**할 수 있게 한다. 모두 서버 없이 단일 HTML에서 구현한다.

### localStorage — 쓴다 (단, 무엇을 저장하느냐가 핵심)

`localStorage`면 충분하다(용량 ~5MB, 텍스트 위치 정보엔 차고 넘침). 단 두 가지 원칙:

- **본문 텍스트가 아니라 위치 메타만 저장한다.** segment id와 오프셋만 담는다(저작물 본문을 저장소에 복제하지 않음, 용량도 절약).
- **오프셋은 "번역 본문(한글) 기준"으로 잡는다.** 빌드가 첫 등장 병기 `(blindsight)`나 한자 `(位相)`, 용어 링크 span 등을 본문에 끼워 넣으면 렌더된 DOM의 텍스트가 달라진다. 그래서 하이라이트 오프셋을 *렌더 결과*에 매기면 표현을 바꿀 때 어긋난다. 대신 segment의 **정본 한글 텍스트(번역 `runs`의 한글만 이어 붙인 문자열)** 기준으로 오프셋을 잡고, 빌드는 병기·링크 span에 `data-anno`(또는 `aria-hidden`성 표식)를 달아 **오프셋 계산에서 제외**한다. 이러면 §1의 분리 원칙대로 표현이 바뀌어도 하이라이트가 살아남는다.

저장 스키마(키는 책 버전으로 네임스페이스):

```js
// blindsight:v1:progress
{ "lastSegment": "s0412", "ts": 1719500000 }

// blindsight:v1:bookmarks
[ { "seg": "s0188", "label": "사라스티 첫 등장", "ts": ... } ]

// blindsight:v1:highlights
[ { "seg": "s0412", "start": 4, "end": 11, "color": "yellow", "note": "" } ]
```

- **이어 읽기**: `IntersectionObserver`로 화면 최상단에 보이는 segment id를 추적해 `progress`에 저장(쓰로틀). 다음 방문 시 그 앵커로 스크롤 복원.
- **북마크**: segment id + 라벨 목록. 목차 옆 패널에서 점프.
- **하이라이트**: 선택 영역을 `{seg, start, end}`(정본 텍스트 오프셋)로 저장. 로드 시 해당 segment의 텍스트 노드를 걸어가며 `data-anno`를 건너뛰고 오프셋을 매핑해 `<mark>`로 재적용.

한계: localStorage는 **기기·브라우저별**이라 기기 간 동기화는 안 된다. 동기화가 필요하면 서버 대신 **내보내기/가져오기**(위 JSON을 파일/클립보드로 export·import)를 제공하면 무서버로 해결된다.

### URL fragment로 위치 공유 — 된다 (segment id 사용)

segment id가 불변(§2.1)이라 **그대로 URL 앵커로 쓴다.** 빌드는 각 segment를 `<… id="s0412">`로 렌더.

- **한 지점 공유**: `reader.html#s0412`. 문단 호버 시 나오는 "🔗" 같은 작은 핸들이 이 URL을 복사. 받는 사람은 열면 그 문단으로 스크롤 + 잠깐 강조.
- **범위 공유**: `reader.html#s0412-s0420`(문단 범위). 로드 시 그 구간을 묶어 강조.
- **선택 영역(인용) 공유**: `reader.html#s0412.4-11` 처럼 정본 오프셋을 붙여, 받는 사람도 같은 글자 범위가 강조된 채 열리게 한다. 하이라이트와 동일한 오프셋 모델을 쓰므로 localStorage 없이도 링크만으로 재현된다.
- 로드 시 fragment를 파싱해 스크롤·강조. id가 빌드 간 불변이므로 **표현을 다시 빌드해도 기존 공유 링크가 계속 유효**하다.

> 문단 "번호"를 노출할지: segment id(`s0412`)는 안정적이지만 사람이 외우긴 어렵다. 독자에게는 부·장면 기반의 읽기 좋은 표식(예: `1부 · 7장면`)을 보여주고, 공유 URL에는 내부 id를 쓰는 2층 구조가 깔끔하다.
